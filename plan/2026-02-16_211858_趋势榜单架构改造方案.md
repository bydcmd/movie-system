# 趋势榜单架构改造方案

生成时间：2026-02-16 21:18:58
文件名：2026-02-16_211858_趋势榜单架构改造方案.md

## 任务背景
当前趋势榜单由 Controller 直连 Service，再由 Service 通过 JDBC 拼接 SQL 查询离线结果表并逐条获取电影详情，存在分层混杂、性能瓶颈与口径不一致等问题。需要整理一份完整的架构改造方案，以提升一致性、性能与可扩展性。

## 方案详情

### 现状分析
- Service 直连 SQL（JdbcTemplate），与业务编排耦合，违背 SRP/DIP。
- 趋势榜单与热门榜单逻辑存在重复与不一致来源。
- N+1 查询明显，limit 较大时性能隐患突出。
- 返回结果缺少 hot_score/calc_date 等口径信息。
- period 参数仅靠字符串手工校验，扩展成本高。
- 索引与查询条件匹配度不足，可能无法命中最优索引。

### 技术方案
- 将榜单查询统一收敛到 Mapper/Repository 层，避免 Service 拼接 SQL。
- 通过一次 JOIN 或批量 IN 查询解决 N+1。
- 引入榜单 DTO，包含 movie + hot_score + rank + calc_date + period。
- period 参数强类型化（enum + 校验）。
- 优化索引或 SQL，使 period_type + calc_date + hot_score 查询高效。
- 明确离线数据延迟/缺失策略，并返回 calc_date 供前端解释。

### 影响范围
- Controller：趋势榜单接口返回结构可能调整。
- Service：`AnalyticsServiceImpl` 查询实现重构。
- Mapper：新增/调整榜单查询 SQL。
- 数据库：索引调整（若执行）。
- 前端：若依赖 `List<Movie>` 结构，需要适配 DTO。

## 原子步骤清单

### 步骤 1：统一榜单查询入口
- **操作对象**：`backend/src/main/java/com/movie/backend/service/impl/AnalyticsServiceImpl.java`
- **具体动作**：移除 JdbcTemplate 拼 SQL，改为调用 Mapper 统一查询接口
- **预期结果**：查询路径统一，逻辑集中便于维护
- **关键里程碑**：是

### 步骤 2：新增/调整 Mapper 查询
- **操作对象**：`backend/src/main/resources/mapper/MovieMapper.xml`、`backend/src/main/java/com/movie/backend/mapper/MovieMapper.java`
- **具体动作**：新增 `selectTrendingWithScore(period, limit)`，JOIN movies 与 stats_hot_movies
- **预期结果**：一次查询拿到所需字段与排序
- **关键里程碑**：是

### 步骤 3：引入趋势榜单 DTO
- **操作对象**：`backend/src/main/java/com/movie/backend/dto`（新增 DTO）
- **具体动作**：设计返回结构：movie 基础信息 + hot_score + rank + calc_date + period
- **预期结果**：接口结果具备口径信息与可解释性
- **关键里程碑**：是

### 步骤 4：参数强类型化
- **操作对象**：`backend/src/main/java/com/movie/backend/controller/AnalyticsController.java`
- **具体动作**：引入 `TrendPeriod` enum + 校验注解/转换
- **预期结果**：参数约束一致，减少手工校验
- **关键里程碑**：否

### 步骤 5：索引/查询优化与口径治理
- **操作对象**：`backend/sql/movie_db.sql`（或迁移脚本）
- **具体动作**：调整索引 `(period_type, calc_date, hot_score)` 或优化 SQL 使用现有索引
- **预期结果**：查询稳定高效
- **关键里程碑**：否

## 预期结果
- 趋势榜单接口性能提升，避免 N+1。
- 口径统一且可解释，前端可展示热度和计算日期。
- 架构分层清晰，后续新增榜单或周期更容易扩展。
